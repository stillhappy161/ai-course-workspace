# 审核审批智能体——完整产品设计

## 一、产品定位

一个**嵌入企业现有OA审批流中的AI审核节点**。不替换OA——只在审批流里插入一个节点，接管必要的审核工作。

**一句话**："我们不换你的OA。我们只是在你审批流里加一个AI节点——审得比人快、比人细。人花一分钟的驳回，AI记一辈子。三个月后，AI比你的财务更了解你公司的审批雷区。"

## 三、审批流程——四层嵌套结构

```
企业
 └── 流程A：费用报销
 │    ├── 节点1：部门主管审批
 │    │    ├── 审批人：申请人.上级（动态路由）
 │    │    ├── 规则1：金额<5000→自动通过
 │    │    ├── 规则2：本人不能批自己→自动跳上级
 │    │    └── 权限：可批/可驳/可转/可看全金额/48h超时
 │    ├── 节点2：财务审核
 │    │    ├── 审批人：财务部.费用审核员（固定角色）
 │    │    ├── 规则1：发票验真失败→自动驳回
 │    │    ├── 规则2：供应商不在白名单→强制人工
 │    │    └── 条件路由：金额>20000→触发节点3
 │    └── 节点3：CFO审批（条件触发）
 │         ├── 审批人：CFO（固定角色）
 │         └── 触发条件：金额>20000 或 类型=招待费
 │
 └── 流程B：采购审批
      ├── ...
```

## 四、流程导入——三条路径

### 路径一：OA有API——直接拉取
泛微、致远、钉钉、企微各写一个适配器→拉取流程定义→转为四层嵌套JSON

### 路径二：OA无API——RPA抓屏
实在Agent登录OA→识别流程→自动导出结构化数据

### 路径三：从历史审批记录反向推断
AI分析5000条历史审批记录→推断流程结构和审批规则→推送管理员确认

### 新建流程——AI一句话自动构建
输入"流程名称+流程描述"→AI自动识别节点、路由条件、生成初始规则、搜索制度文件补充规则、从历史案例推断规则→推送管理员审核

## 五、数据取数——审批单触发后要查什么

```
OA推送审批单
    ↓
① 单据本身的数据（金额/类别/申请人/部门/附件）
② 企业地图数据（申请人上级/KPI/区域）——<50ms内存查询
③ ERP/财务系统实时数据（供应商历史/预算余额/历史价格）——超时5秒兜底
④ 语义层定义（"超标"怎么定义、"招待费"包含什么）——内存查询
⑤ 历史审批记录（申请人/供应商/部门通过率统计）——本地DB查询
```

### 取数后的分析

**硬规则匹配**：规则引擎接收的不是原始数据——是已经计算好的变量：
```python
context = {
    "amount": 4200,
    "supplier_in_whitelist": False,       # 来源：ERP API
    "budget_remaining": 12800,            # 来源：ERP API
    "applicant_rejection_rate_6m": 0.23,  # 来源：历史统计
    "price_vs_historical_avg": 1.12,      # 来源：ERP查历史价格→对比
    "supplier_first_time": True,          # 来源：ERP查是否有过交易
}
```

**软判断准备**：从数据里找"反常"。不是判断对错——是找"值得老板看一眼的东西"：
- 价格异常→打标签
- 新供应商→打标签
- 申请人历史驳回率高→打标签
- 预算紧张→打标签
- 方案质量→LLM语义分析→打标签

## 六、硬规则 vs 软判断——两套处理逻辑

| | 硬规则 | 软判断 |
|---|---|---|
| **谁来定** | 制度文件+历史模式 | 没有制度。老板的品味和判断 |
| **AI做什么** | 条件匹配→自动执行 | 准备决策简报+打标签+记住老板习惯 |
| **人做什么** | 确认AI的规则建议 | 判断。AI不做判断 |
| **蒸馏什么** | 条件→动作 | 习惯→优先级。标签→提醒 |
| **存储在哪** | rules.json | 小像（七维模型的信息偏好维度） |

**原则**：AI可以猜老板会怎么判断。但AI不能替老板判断。AI让老板判断的时候更快。

## 七、小像在审批中的作用

小像不只影响AI"怎么说"给老板听——**影响AI判断的三个维度**：

### 小像七个维度在审批中的映射

| 小像维度 | 改变什么 | 例子 |
|---|---|---|
| **信息偏好** | 卡片展示策略。事实型→全部展开。模式型→只展示关键一条 | 同一笔审批单，两个老板看到的卡片不同 |
| **判断基准** | 归因方向。逻辑型→归因到流程。人际型→归因到人 | "刘强驳回率高"→流程问题 vs 能力问题 |
| **行动阈值** | 什么算异常。进攻型→8%预警。防守型→15%才预警 | 同一个价格偏离，触发不同的预警等级 |
| **时间视野** | 看多远。短期→这个月预算。长期→三年总成本 | 审批分析维度不同 |
| **认知模式** | 找几层因果。线性→一根链。系统→一张网 | 信号之间是否关联展示 |
| **深层价值观** | 优先级排序。增长/成本/人/系统——不同的老板看到不同的优先级 | 价格异常对成本敏感型是高风险，对增长型是中等 |
| **文化模因** | 基准线。制造出身→成本偏差3%正常。贸易出身→利润率波动是核心 | 同一数字不同的解读方式 |

### 小像参数注入软判断引擎
- 不改变客观数据
- 改变：阈值（什么算异常）、权重（什么信号更重要）、归因方向（问题指向流程还是人）

## 八、小像的持续学习

### 三层反馈机制

**第一层：直接反馈**——老板点了通过/驳回、写了什么理由
- 老板驳回时写了详细技术层面的理由→小像向"事实驱动"方向微调
- 老板确认了AI的规则建议→小像向"防守型"方向微调

**第二层：间接反馈**——老板的行为在说话
- 看卡片时长→偏好模式驱动还是事实驱动
- 跳过/从未点开的信号→降低该信号权重
- 主动追问的信息→下次该信息展示在第一屏

**第三层：模式对比**——通过/驳回的统计分析
- 取最近100单→对比通过组和驳回组的特征差异→自动校准阈值

### 两种更新节奏
| 参数 | 更新频率 |
|---|---|
| 信息偏好 | 每10-20单微调 |
| 行动阈值 | 每20-30单微调 |
| 信号权重 | 每10单校准 |
| 追问学习 | **即时更新** |
| 监管模式 | 每周报告→老板手动确认/修正 |

## 九、规则蒸馏——三源合一

### 源一：人的反馈（被动学习）
李会计驳回→AI记录原因→生成规则建议→推送确认

### 源二：制度文件（主动学习）
上传制度PDF→AI提取规则→发现冲突→标出冲突→人判对错

### 源三：历史案例（主动学习）
分析500条历史记录→发现"哪些特征组合大概率会通过/驳回"→生成置信度→推送确认

### 蒸馏后的规则统一写入 rules.json，带 source 标记
```json
{
  "rule_id": "RULE_017",
  "name": "XX五金+张三+小额=仅通知",
  "condition": "supplier == 'XX五金' and applicant == '张三' and amount < 3000",
  "action": "notify_only",
  "source": {
    "type": "ai_discovered",
    "method": "history_analysis",
    "confidence": 0.92,
    "confirmed_by": "管理员张三"
  }
}
```

## 十、操作台——五个核心模块

| 模块 | 做什么 | 对应前面讨论 |
|---|---|---|
| **流程管理** | 流程列表+可视化编辑+一键导入+AI新建 | 四层嵌套结构 |
| **规则管理** | 可视化规则编辑器。每条规则带开关、优先级、来源 | rules.json的操作界面 |

| **实时监控** | 今日审批统计+异常告警+系统健康 | 包装层 |
| **审计追溯** | 按时间/人/金额/结果筛选追溯 | 合规要求 |
| **语义层管理** | 审批中发现的未定义术语→人补充→反写语义层 | 语义层进化 |

## 十一、老板交互——企微/钉钉/飞书聊天框

老板不需要打开操作台。

### 交互一：日常审批——卡片+按钮+语音回复
企微卡片→点通过/驳回→或语音回复→AI语音转文本→解析意图→记录审批意见

### 交互二：追问——老板想了解更多
老板语音："这个供应商上次跟谁合作过？"→AI10秒内回复→追问=小像学习信号

### 交互三：定期复盘——每周总结推送
"本周你做了23个审批决定。系统观察到：你对价格异常的容忍度在提高。"→老板可以聊、可以修正

### 交互四：规则蒸馏确认——简化版
"新规则建议：新供应商+报价偏离>10%→标注高风险。[好，加上这条规则] [不用]"
