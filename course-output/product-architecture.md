## 架构说明

### 数据接入层
企业的原始数据散落在ERP、CRM、MES、OA及非结构化文件中。数据接入管道通过`config.json`字段映射（如"业务员"→EMP037），在数据写入时自动打上企业地图坐标。

### 底座层（共享）
两个产品共享同一个三层底座：

- **企业地图**（第一层）：公司长什么样——对象（10大类）、关系（6类）、规则（3类）。AI不猜，沿着地图走
- **语义定义**（第二层）：公司怎么说——术语怎么定义、指标怎么算、数据从哪取、更新怎么管。版本化+fallback自动调参
- **小像**（第三层）：老板怎么想——七维决策画像。慢收敛=高壁垒

**事件映射引擎**串联三层：每一笔ERP数据在写入时自动标注"发生在哪（地图）、按什么口径（语义）、应该怎么告诉老板（小像）"。

### 产品层
两个产品共享底座，互补形成完整闭环：

- **产品一（经营决策）**：老板看到全局——发生了什么、为什么、该找谁
- **产品二（审核审批）**：一线业务执行——审批智能体嵌入OA，规则蒸馏让产品越用越聪明。审批中产生的新规则回写到底座→老板在经营决策平台看到"华东区超标已审批"

### 用户触点层
不建独立APP。嵌入企业已有平台——企微/钉钉推送、独立Web看板、API对接第三方OA。

## 技术栈

| 层 | 核心组件 | 技术选型 |
|---|---|---|
| **用户触点** | 消息推送、Web看板 | 企微/钉钉API + React |
| **产品层** | 经营分析、审批智能体 | Dify工作流 + LangGraph补充 |
| **底座层** | 三层定义文件 | JSON/YAML，内存遍历，<50ms |
| **事件映射** | 数据打坐标、走地图 | Python + events.jsonl |
| **数据接入** | 字段映射 | config.json + ERP/CRM API |
| **大模型** | 推理+判断 | DeepSeek V4(主力) + Claude Sonnet 5(审批) |
| **部署** | 私有化/SaaS | Docker + SQLite(单机) / PostgreSQL(多租户) |

## 对政府的说法

企业数字孪生基础设施 + AI决策智能平台 + 中小企业AI赋能公共服务平台

## 对投资人的说法

三层底座=单客户LTV持续递增（越用越准→迁移成本越高）+ 双产品=客户钱包份额叠加（经营决策+审批智能体）+ 四波竞品无直接竞品（蓝海定位）+ 咨询基因=技术护城河之外的第二层壁垒（不可复制）
